# 주석 리팩터링 설계

**날짜:** 2026-04-13
**대상:** `app/` 및 `tests/` 내 모든 `.py` 파일

---

## 목표

비유 중심의 과도한 주석을 제거하고, 기능적 주석만 간결하게 유지한다.
코드 자체가 읽히도록 주석 밀도를 낮추되, 비자명한 개념은 짧게 설명을 유지한다.

---

## 처리 방식

**접근법 A — 파일 단위 순차 처리**
파일 하나씩 수정 → 각 파일 완료 후 `pytest tests/ -v` 로 회귀 확인.

처리 순서:
1. `app/main.py`
2. `app/core/config.py`
3. `app/schemas/ocr.py`
4. `app/api/routes/ocr.py`
5. `app/services/extractor.py`
6. `tests/test_ocr.py`

---

## 적용 규칙

### 제거

| 대상 | 예시 |
|------|------|
| `┌──...┐` 박스형 비유 블록 | `┌─ 비유: 음식점의 홀 직원 ─┐` |
| `# 비유:` 로 시작하는 줄 | `# 비유: 냉장고 문을 열고...` |
| `[읽기 순서 N/7]` 파일 헤더 docstring 전체 | 파일 상단 긴 역할 설명 블록 |
| import 아래 3~5줄짜리 설명 블록 | `# pdfplumber : PDF에서...적합. 스캔한 PDF는...` |

### 교체

| 변경 전 | 변경 후 |
|---------|---------|
| import 아래 여러 줄 설명 | import 오른쪽 한 줄 인라인 주석 |
| 함수 위 긴 역할 설명 docstring | 역할 한두 줄 + Args/Returns (필요 시) |
| 섹션 구분 긴 박스 헤더 | `# --- 섹션명 ---` 한 줄 |

### 유지

| 대상 | 이유 |
|------|------|
| `[변경 이력]` 블록 | 프로젝트 히스토리 추적 |
| 비자명한 로직 인라인 주석 | 코드만으로 의도 파악 불가 |
| 아래 "어려운 개념" 설명 | 삭제 시 이해 불가 |

### 어려운 개념 — 짧게 유지

| 개념 | 유지할 설명 |
|------|-------------|
| `Laplacian 분산` | 엣지 분산으로 선명도 측정. 높을수록 선명. |
| `Otsu 이진화 (THRESH_OTSU)` | 흑/백 분리 임계값을 히스토그램으로 자동 계산. |
| `OMP_NUM_THREADS=1` | ThreadPoolExecutor와 Tesseract 내부 스레드 경쟁 방지. |
| `default_factory=list` | dataclass에서 `= []` 쓰면 모든 인스턴스가 같은 리스트를 공유함. |
| `image_to_data()` vs `image_to_string()` | data는 단어별 신뢰도 포함, string은 미포함. |
| `conf == -1` 필터 | Tesseract가 텍스트 외 영역(구분선 등)에 -1을 반환. |
